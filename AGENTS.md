## Common gotchas

- **Glossary ¬ß9 disputed pairs matter in code.** DB: `rejected`. UI label: `Decline` / `Declined`. Variable names in `application/` should match the DB term (`status: 'rejected'`); UI strings use the UI term. See [docs/spec/_translation-glossary.md](./docs/spec/_translation-glossary.md) ¬ß9.
- **`my_status` is NOT `JoinRequest.status`.** `my_status` is a UI-derived enum that lives only in poll payloads. Mapping table is in [global.md ‚Üí "Polling sync"](./docs/spec/pitchup-spec-global.md).
- **Match status is computed on-read.** No `status` column on `Match`. Derived from `start_time`, `duration`, `cancelled_at`. Single helper, never duplicate the logic.
- **Slot math has a single canonical formula.** `computeSlots(match)`. Never recompute locally ‚Äî see [global.md ‚Üí "Slot math"](./docs/spec/pitchup-spec-global.md).
- **Captain cannot Join or Watch their own match.** Both endpoints explicitly check `user !== match.captain_id` (400 codes). UI doesn't show the buttons, but backend backstop is mandatory.
- **`Match.cover_id` is a snapshot at INSERT.** Changing `venue.cover_id` later does not propagate.
- **Author resolution at render-time, not write-time.** Chat messages from banned/deleted users render as `[Removed user]` retroactively without migration.
- **Material vs non-material edits.** Editing `start_time`/`duration`/`venue`/`surface`/`studs_allowed`/`price`/`field_booked` notifies accepted players. Editing `total_spots`/`captain_crew`/`description` is silent ‚Äî **EXCEPT** `total_spots ‚Üë` may trigger `notify watching` (separate channel).
- **Two poll endpoints, both at 15s foreground / 60s background.** Don't write a third.
- **Spec wins over app-map.** If [docs/spec/pitchup-app-map.md](./docs/spec/pitchup-app-map.md) and a `docs/spec/*.md` file disagree, the spec file is right and the map needs a fix.
- **Russian archive (`docs/spec/ru/`) is frozen.** Do not read it without explicit human instruction. See [docs/spec/CLAUDE.md](./docs/spec/CLAUDE.md).
- **Don't set `font-size` on `html` / `body`.** Tailwind's rem-based sizing (`h-12` ‚Üí 48px, `h-11` ‚Üí 44px) assumes the browser default 16px root. Setting `html { font-size: 15px }` silently shrinks every rem-based class (Layer 0 ran into this: 48px buttons became 45px). Body text size is applied via Tailwind utility (`text-[15px]`) on the `<body>` element in `app/layout.tsx` instead.
- **`pnpm` may not be on PATH.** On the current dev machine corepack is absent and no global `pnpm` is installed. Run commands as `npx pnpm@9.15.4 <args>` (slower per-call but no setup), or have the human enable corepack once. Don't fall back to `npm install` ‚Äî it would rewrite the lockfile.
- **Prisma is pinned to 6.x, not 7.x.** Prisma 7 removed `url = env(...)` from `schema.prisma` and requires `prisma.config.ts` + a driver adapter. Our scaffolding (ARCHITECTURE.md ¬ß8, repo `src/shared/db/prisma.ts`) assumes classic Prisma. If you ever bump to 7 ‚Äî open ADR-0004 first; do not silently upgrade.
- **`prisma` CLI does not auto-load `.env.local`.** Next.js reads it; the Prisma CLI only reads `.env`. To run `prisma migrate deploy` / `migrate status` / `generate` against the dev DB, export `DATABASE_URL` explicitly first: `export DATABASE_URL=$(grep '^DATABASE_URL=' .env.local | cut -d= -f2-) && pnpm exec prisma migrate deploy`. The value contains `&` (Neon query string) ‚Äî extract it as a `$(...)` value, don't `source` the file naively (bash chokes on the unquoted `&`).
- **Use `prisma migrate deploy`, never `migrate dev`, against the shared Neon DB.** `migrate dev` requires a shadow database and may prompt destructive resets. `deploy` only applies pending migrations idempotently. `migrate reset` drops the entire `public` schema ‚Äî never run it without explicit user confirmation, the DB is shared.
- **Don't recompute slot math or match status in handlers / components.** There is one canonical source for each: `computeSlots(match, acceptedSlots)` in [`src/match_lifecycle/domain/slot-math.ts`](./src/match_lifecycle/domain/slot-math.ts) and `deriveMatchStatus(match, slots, now)` in [`src/match_lifecycle/domain/match-status.ts`](./src/match_lifecycle/domain/match-status.ts). The application layer calls them once and passes the result downstream. If you find yourself writing `match.totalSpots - ‚Ä¶` or `now > match.startTime` anywhere else, stop ‚Äî call the helper.
- **Seed data lives inside a migration file** (`prisma/migrations/*_seed_demo_data/migration.sql`), not in a `prisma/seed.ts` script. Seed rows use relative times (`NOW() + INTERVAL '‚Ä¶'`) so the dataset stays evergreen no matter when the migration is applied. Seed captains use a deliberately fake `google_sub` (`seed:captain-N`) and `.invalid` email so they cannot collide with a real OAuth user.
- **Funnel every "by day" / horizon filter through `src/shared/time/prague.ts`.** `pragueDay(date)` / `pragueRange(d1, d2)` / `todayPrague(now)` are the canonical primitives (see [spec global.md ‚Üí Timezones & date ranges](./docs/spec/pitchup-spec-global.md)). Naive `BETWEEN start_of_day_utc AND start_of_day_utc + INTERVAL '24h'` silently drops or duplicates an hour on the two DST Sundays ‚Äî that's the bug these helpers prevent. Layer 2.5 spec-tested against 2026-03-29 (23h) and 2026-10-25 (25h).
- **Discover URL params are whitelist-parsed and never throw.** `parseDiscoverFilters` (in `src/match_lifecycle/application/discover-filters.ts`) silently falls back to `today_prague(now)` for malformed/past/>+20 `?date=`, drops unknown enum values for `?distance/time/size/spots`, returns `null` cursor on decode failure. **Don't add `400` responses for bad query strings** ‚Äî the spec explicitly requires graceful fallback ("Invalid query params fallback"). Same parser is shared by `/games` (Server Component) and `GET /api/matches/discover` (Route Handler).
- **`?distance=` has client-only truth.** The saved location lives in `localStorage` (`pitchup.location`, see `app/(public)/games/use-saved-location.ts`). SSR can't see it ‚Äî the service silently drops the distance filter on first paint, and the `DistanceBanner` client island plus a one-shot client refetch take over after hydration if a location is present. Don't try to read the location server-side, and don't 4xx on `?distance=` without it.
- **Filter-changing URL writes use `router.replace`, not `push`.** Day-picker selection and filter-sheet Apply should not pollute browser history ‚Äî back-button should leave `/games`, not walk through filter combinations. Search + Show more don't touch the URL at all (search is ephemeral per spec; pagination is a fetch).
- **Discover pagination is keyset on `(start_time ASC, id ASC)` with a `LIMIT n+1` trick.** The Prisma adapter (`prisma-match-repository.ts`) takes one extra row to detect `nextCursor` without a second query. Cursor format is `base64url(JSON{s,i})` ‚Äî opaque to clients, kept tight via `encodeCursor`/`decodeCursor` in `discover-filters.ts`. **Don't recompute cursor format elsewhere** ‚Äî round-trip through these helpers.
- **Discover SQL uses `$queryRaw`, not Prisma's `where` builder.** Three of the filters can't be expressed via Prisma's typed API: Prague-TZ hour extract (`EXTRACT(HOUR FROM start_time AT TIME ZONE 'Europe/Prague')`), Haversine distance, and tuple cursor comparison `(start_time, id) > ($1, $2)`. The handwritten query is in `prisma-match-repository.ts`; ILIKE search escapes `%` and `_` to neutralise wildcard injection. Keep the formula for `free_slots` (`total_spots - 1 - COALESCE(array_length(captain_crew, 1), 0)`) **in sync with `computeSlots()`** when Layer 4 adds the accepted-requests JOIN.
- **Create-match has no advisory lock (and that is correct).** `POST /api/matches` is the *only* match-mutating endpoint that does **not** wrap its work in `withMatchLock` ‚Äî the match id doesn't exist until the row is inserted, so there's nothing to lock on. The "Per-endpoint checklist" in match.md spells this out as an explicit exception; do not add a lock here when copying the Join/Approve/Kick patterns later. The other no-lock exception is `POST /api/matches/:id/messages` (chat ‚Äî no slot/status mutation).
- **`Match.coverId` is taken from the venue at INSERT, never afterwards.** `CreateMatchService` reads `venue.coverId` and passes it into `matchRepository.create({...})` as a frozen snapshot. The Prisma schema's `Match.coverId` column has no FK to venues ‚Äî it's a denormalised slug. Changing the venue's cover later does not propagate (spec / global.md "Cover venue"). Don't try to "fix" this with a JOIN.
- **Wizard error toasts dispatch on the backend `code`, not status.** `app/(private)/matches/new/wizard.tsx#handleBackendError` switches on `body.code` returned by `toHttpResponse`. When you add a new validation error to `match_lifecycle/domain/errors.ts`, also add a `case` here ‚Äî otherwise the user falls through to "Couldn't publish. Try again." which buries the real reason.
- **Discover bottom-sheet is hand-rolled (no Radix Dialog yet).** Backdrop click + Esc close + body-scroll lock are implemented inline in `more-filters-sheet.tsx`; focus-trap intentionally deferred. Layer 5 captain-sheet will be the second use case ‚Äî extract a shared `Sheet` primitive then (and consider adding `@radix-ui/react-dialog` at that point; same family as the existing Radix Checkbox/Switch).
- **`MatchRepository.findById(id, tx?)` is the only read method, no `findByIdForUpdate`.** Pass the `tx` from `withMatchLock` to read under the advisory lock; omit it for unlocked reads. `SELECT ... FOR UPDATE` would be theatre on top of `pg_advisory_xact_lock` (spec / ARCHITECTURE.md ¬ß8 explicitly forbid it). When growing the port in later layers, follow the same one-method-with-optional-tx shape.
- **JoinRequest UPSERT is a SELECT-then-(INSERT|UPDATE) sequence, safe under the advisory lock.** `PrismaJoinRequestRepository.upsertToPending` returns a discriminated `{outcome: 'inserted' | 'revived' | 'conflict'}` instead of throwing on a status conflict ‚Äî the service maps `conflict.existingStatus` (`pending` ‚Üí `AlreadyRequestedError`, `accepted` ‚Üí `AlreadyInMatchError`). Don't reach for a Postgres `ON CONFLICT` here: the discriminated-outcome shape carries the existing status, which the upsert needs to choose the matching domain error; raw upserts can't surface that.
- **Notification aggregate is deliberately absent in Layer 4.** Approve / reject services have `// TODO(Layer 7): notification(...)` markers at the insertion points (inside the locked tx, before commit ‚Äî spec match.md "Write ordering"). When Layer 7 lands, inject a `NotificationRepository` port via the constructor and add one line per marker. **Do not write a stub port now** ‚Äî we'd have to throw it away once the real `Notification` shape (type enum, body templates) is known.
- **Unit tests mock `withMatchLock` via `vi.mock`** to bypass the real Prisma transaction. Fake repositories ignore the sentinel `tx` argument. Race scenarios are simulated by calling the service twice sequentially against a shared fake (the second call sees the state the first one wrote). Real concurrency belongs to integration tests against Postgres, which Layer 4 does not yet ship.
- **Route Handlers under `app/api/matches/[id]/...` use Next 15 async `params`.** Signature: `{ params }: { params: Promise<{ id: string }> }`, then `const { id } = await params;`. Older Next 14 sync-params will type-check but warn at runtime ‚Äî always `await` it.
- **Chat writes are the second no-lock exception (alongside `POST /matches`).** `POST /api/matches/:id/messages` and `DELETE /api/matches/:id/messages/:msgId` do NOT wrap in `withMatchLock` ‚Äî chat doesn't mutate slot/status/roster, spec match.md ¬ß546 says timestamp ordering is sufficient. Adding a lock here would be theatre. The cross-match guard on delete (verifying `message.matchId === url:id`) is the only safety check needed.
- **Layer 4 ports now accept optional `tx` on reads.** `JoinRequestRepository.findByMatchAndUser` / `findById` / `listAcceptedForMatch` + new `listPendingForMatch`, and `WatchRepository.countForMatch` / `existsForUserAndMatch`, are all callable with NO `tx` ‚Äî they fall back to the singleton prisma in the adapter constructor. The `tx`-passed call sites in Layer 4 still operate under the advisory lock; the no-`tx` call sites in Layer 5 (chat role gating, `MatchStateService`, viewer-role derivation, RSC page load) read against the unlocked client. Same shape as `MatchRepository.findById(id, tx?)`. When extending other ports in later layers, mirror this convention ‚Äî never duplicate methods for "locked" and "unlocked" variants.
- **`MatchStateService` is a cross-context read-model assembler.** Lives in `match_lifecycle/application` but imports from three contexts (`chat/domain/chat-message-repository`, `auth/domain/user-repository`, `match_lifecycle/domain/*`). Application-layer composition across contexts is allowed ‚Äî only `domain/` is forbidden from cross-context imports. The service does NOT enforce the polling-membership 403 (`chat_forbidden` for pending / watching / guest); that gate lives in the `GET /api/matches/[id]/state` route handler so the RSC page can call the same service for everyone. Layer 6+ services that need to assemble a similar cross-context view should follow this pattern ‚Äî service is pure read, gate lives at the interfaces edge.
- **Polling endpoint never throws on bad `?since=`.** `GET /api/matches/[id]/state?since=foo` parses leniently ‚Äî invalid ISO ‚Üí `null` ‚Üí full history. Same convention as Discover's whitelist URL parsing ("never 4xx on bad query strings"). Adding a 400 would force the client into a panic state on garbage URLs that should just degrade gracefully.
- **`computeCta(matchStatus, viewerRole, isFull)` is the SINGLE source of truth for the CTA bar.** Pure function in `match_lifecycle/domain/compute-cta.ts`, table-driven test in `tests/match_lifecycle/domain/compute-cta.test.ts` covers every cell of the spec ¬ß77-103 cascade. UI never inlines CTA logic; if a new role or status branch needs to render differently, change the function and add the row to the test ‚Äî both update atomically. Out-of-scope Layer 6 actions are flagged `comingSoon: true` and render disabled with a "Coming soon" tooltip ‚Äî when Layer 6 lands, just flip the flags and wire the click handlers; the cascade itself doesn't need to change.
- **Polling cadence is gated on viewer role at the hook level, not the endpoint.** `MatchShell` passes `enabled: isLiveStatus && (role === captain || accepted)` to `usePolling`. Pending / watching / guest see the initial static snapshot from the RSC and never poll (spec ¬ß215-216). Don't add a "kind, sort of polling" mode for watchers ‚Äî they get the in-app inbox `spot_opened` signal in Layer 7 instead, exactly as the spec prescribes.
- **`optionalAuth()` returns `null` for not-yet-onboarded users.** Same DB-checked invalidation as `requireAuth` (banned / deletedAt / missing User row ‚Üí null), but does not throw. Use it in Server Components that must serve guests AND signed-in users without redirecting (currently `/matches/:id`; Layer 6 `/me` will need the throwing variant since `/me` requires onboarding). Never trust the JWT alone; the row check is mandatory.
- **`Match.cancelReasonHidden` is honoured at the RSC layer, not in the domain.** The DTO surfaced to the page passes `match.cancelReason` straight through; the page itself flips it to `null` when `cancelReasonHidden === true`. Same pattern as `description` / `descriptionHidden`. Don't filter inside the application service ‚Äî moderation is a presentation concern.
- **Realtime chat (Ably) shipped in Layer 5.5 ‚Äî the `// TODO(Layer 5.5)` markers are now REAL fan-out (ADR-0005).** `PostChatMessageService` / `DeleteChatMessageService` take a `ChatRealtimePublisher` port (`src/chat/domain/chat-realtime-publisher.ts`) and publish AFTER persistence, **best-effort** (the publish is wrapped in `try/catch` + `console.error` IN THE SERVICE ‚Äî mirrors the ADR-0004 post-commit email send; a transport hiccup never fails the 200, polling delivers within 15s). Two adapters chosen by key presence like `emailSender`: `AblyChatRealtimePublisher` (`Ably.Rest`, `ABLY_API_KEY`) when the key is set, else `NoopChatRealtimePublisher` (silent ‚Äî realtime is an enhancement, NOT a required channel, so a missing key is NOT a startup error, unlike `EMAIL_TRANSPORT=resend`). Polling stays the source of truth (spec ¬ß229).
- **Ably event payloads carry `author_id` (raw), NOT the resolved author object.** `message_created` = `{ id, author_id, text, created_at }`, `message_deleted` = `{ id, deleted_at }` (spec ¬ß235-236; the canonical shapes + the shared `chatChannelName()` + `CHAT_REALTIME_EVENTS` live in the domain port, used by BOTH the server adapter and the client hook). The client (`match-shell.tsx` ‚Üí `resolveAuthorFromLineup`) resolves `author_id` against the lineup snapshot it already holds (captain + accepted players); an unresolvable id (just-approved player posting before the lineup updates) renders `[Removed user]` for one poll cycle until reconciliation. Do NOT add an author read inside the publish path ‚Äî that's the trade the spec made.
- **Who SUBSCRIBES to Ably is wider than who POLLS.** `useAblyChannel` (`src/ui/hooks/use-ably-channel.ts`) is gated `activeTab === 'chat' && role !== 'pending'` ‚Üí captain + accepted + **watching + guest** subscribe (spec ¬ß244); pending don't. Polling stays gated to captain + accepted only (`MatchShell`'s `canPoll`). Don't conflate the two gates. The hook dynamically `import("ably")` INSIDE its effect (browser-only, same pattern as MapLibre ¬ß155), reads `NEXT_PUBLIC_ABLY_SUBSCRIBE_KEY` via `process.env` directly (NOT `src/shared/config/env`, which is server-only), and no-ops when the key is absent. Gap-fill on reconnect (`connected` after `disconnected`/`suspended`) fires `onReconnect` ‚Üí the shell refetches `GET /state?since=` (same path as polling, ¬ß246). clientId = userId (signed-in) or per-tab `anon-${uuid}` in sessionStorage (¬ß242).
- **`MessageRow` derives `deleted` from the prop, not a one-shot `useState`.** `chat-tab.tsx`: `const deleted = message.deleted_at !== null || locallyDeleted` (the local flag is only the captain's optimistic feedback). Layer 5 had `useState(message.deleted_at !== null)`, which never re-read the prop ‚Äî so a deletion pushed in by ANOTHER user (poll OR realtime) was invisible to non-captain viewers. Layer 5.5 surfaced + fixed this latent bug. If you add another prop-driven mutable flag to a keyed list row, derive it from the prop the same way.
- **`my_status` derivation lives in `match_lifecycle/domain/derive-my-status.ts` (pure), NOT in `application/`.** Same family as `viewer-role.ts` and `compute-cta.ts` ‚Äî pure function from the spec's canonical table (global.md ‚Üí "Polling sync" ‚Üí my_status), table-driven test covers every cell. The application service (`ListMyMatchesService`) and the Layer 7 polling assembler both import it. When Section Past needs the user-facing sub-label ("You left" / "You were removed" / "Request expired" / "Match was cancelled"), read `JoinRequest.status` + `auto_reason` directly ‚Äî NOT through `my_status`. The mapping collapses three different `auto_reason` values into one `declined`, but the sub-label has to distinguish them (spec personal.md sub-label table).
- **`notifyWatching` is a pure helper, not a class ‚Äî designed to be reused by Layer 6.5 Kick + Layer 6.5 Edit-total‚Üë + Layer 7 dispatcher.** Lives in `match_lifecycle/application/notify-watching.ts`. Signature: `notifyWatching(ports, { matchId, slotsBefore, slotsAfter, captainId, triggeredByCaptain, tx })`. Caller is already inside `withMatchLock` and passes the same `tx` ‚Äî the helper acquires NO lock (one advisory lock per transaction is the invariant from ARCHITECTURE.md ¬ß8). The helper short-circuits when `slotsBefore.isFull === slotsAfter.isFull` (false ‚Üí false or true ‚Üí true); only the true ‚Üí false flip captures watcher ids + bulk-deletes via `WatchRepository.deleteAllForMatch`. The `triggeredByCaptain` flag mirrors spec match.md "notify watching" step 4 captain self-trigger skip: `false` for Leave (captain DOES get notified, they didn't free the slot), `true` for Kick / Edit total‚Üë (captain initiated, no self-push). `// TODO(Layer 7)` markers in the body mark exactly where `NotificationRepository.insertMany(spot_opened, ...)` lands.
- **Three no-lock write endpoints, NOT two.** AGENTS used to list two exceptions (chat writes + match creation). Layer 6 added the third: `DELETE /api/matches/:id/watch` (UnwatchMatchService). Per spec match.md "Per-endpoint checklist" ‚Üí DELETE /watch: "idempotent ‚Äî no checks, always 200". DELETE on the composite-PK Watch row is atomic at row level, doesn't mutate slot/status/roster, doesn't trigger `notifyWatching`, and doesn't race with any other endpoint in a way a lock would resolve. The service uses `prisma.$transaction` to satisfy the port's `tx`-typing without actually taking the advisory lock. **Full list of no-lock write endpoints:** `POST /matches` (creation ‚Äî no id to lock on yet) ¬∑ `POST /matches/:id/messages` + `DELETE /api/matches/:id/messages/:msgId` (chat ‚Äî no slot/status mutation) ¬∑ `DELETE /api/matches/:id/watch` (Layer 6 ‚Äî pure idempotent DELETE). Mirror the same pattern when adding similar "no state mutation" endpoints.
- **`MatchRepository.findCaptainMatches(userId)` + `findByIds(ids[])` were added in Layer 6 for `/my-matches`.** Both return `MatchWithVenue` (Prisma `include: { venue: true }`) ‚Äî same shape as Discover, so the existing `MatchCard` UI component consumes them directly. `findByIds` handles empty input as a zero-DB-call short-circuit. When extending the port for similar cross-aggregate reads (e.g. likes view, admin list), keep the "with venue" join ‚Äî `Match` alone is rarely useful for rendering.
- **`/my-matches` Past pagination is in-memory keyset, NOT a separate raw-SQL query.** `ListMyMatchesService.executePastPage({ userId, cursor })` re-fetches the user's complete relationship set (captain matches + JR rows + watch rows + accepted-slot counts), buckets Past, sorts `start_time DESC, id ASC`, applies the cursor predicate, slices. O(N) over the user's full history per page ‚Äî acceptable for a personal page bounded in practice. The SQL-side optimisation (single raw-SQL with cursor in WHERE) lands when history scale demands it; the DTO contract `{ rows, nextCursor }` stays identical. Cursor format reuses Discover's `encodeCursor`/`decodeCursor` ‚Äî `base64url(JSON{s, i})`. Don't introduce a second cursor format.
- **`compute-cta` `comingSoon` flag in Layer 6 only applies to `like` now.** Layer 5 had four `comingSoon` actions; Layer 6 flipped `leave` / `cancelRequest` / `notifyMe` / `stopWatching` to `disabled: false, comingSoon: false` because their backends shipped. Bodies are pure builders `(to, matchUrl) ‚Üí EmailMessage`; they carry a `/matches/:id` deep link (built via `matchUrl(appBaseUrl, id)`) ‚Äî the ONLY interpolation, no venue read.
- **The morning loop catches per-pair send failures so ONE bad recipient can't starve the rest of the run.** `MorningReminderService` wraps each `(match,user)` `withTransaction` in a `try/catch`: on throw the tx rolled back (ledger empty ‚Üí retried next tick), and the catch logs + `continue`s + bumps the new `failed` counter in the result. WITHOUT the catch, a persistently-failing recipient (e.g. one invalid address) would re-throw at the same ordered position every tick and permanently block every recipient after it. The unit tests can't exercise the DB-rollback half (the `withTransaction` mock is a pass-through that doesn't revert the in-memory fakes) ‚Äî that "exactly-once via ledger" property is covered by the ON-CONFLICT test + the Neon smoke run; the unit tests assert the run-level resilience + the gate.

- **Two admin gates, picked by surface (Layer 9a).** `requireAdmin()` (src/auth/composition) is the API gate ‚Äî it builds on `requireAuthCore` and THROWS `ForbiddenError("admin_required", 403)` for a signed-in non-admin; use it in every `app/api/admin/**\/route.ts`. `requireAdminPage()` is the RSC gate ‚Äî it REDIRECTS instead of throwing (guest ‚Üí `/login?callbackUrl=/admin`, non-admin ‚Üí silent `/my-matches`, NO 403 page ‚Äî the spec hides the panel's existence); use it in `app/admin/**` Server Components. Both read `is_admin` from the DB on every call (never the JWT ‚Äî promote/demote take effect with no re-login, spec global.md). The middleware `/admin` rule (silent /my-matches for non-admins, bare `/admin` ‚Üí `/admin/users`) is the FIRST gate; `requireAdminPage` is the in-page backstop. Don't return a 403 page for the admin area ‚Äî that would leak its existence.

- **`BanUserService` is `DeleteAccountService` with a different actor (Layer 9a).** Same three-phase non-atomic cascade: last-admin guard ‚Üí cascade-cancel upcoming captain matches via the UNCHANGED `CancelMatchService` (with `SYSTEM_CANCEL_REASONS.organizerRemoved`) ‚Üí `setBanned(true)` LAST (retry-safe). The admin's ban *reason* goes ONLY to the `admin_actions` audit row ‚Äî players see the canonical organizer-removed wording (privacy: ban must be indistinguishable from self-delete in public UI). `is_admin` is NOT reset on ban (spec global.md ‚Äî a banned admin is simply not counted as active by `countActiveAdmins`). Lives in `src/moderation/application/` (not auth) but composes auth's `UserRepository` + match_lifecycle's `CancelMatchService` at the moderation composition root ‚Äî same cross-context pattern as `DeleteAccountService`.

- **Admin user-management guards live in the services, in this order: self-mod FIRST, then idempotency, then last-admin.** `target===actor` ‚Üí `SelfModificationError(403, self_modification)` before any DB read (spec "Self-modification guard" ‚Äî the `(you)` row mirror). Already-banned / already-admin / not-banned / non-admin ‚Üí idempotent no-op (`applied:false`, NO duplicate audit row). Last-admin (`countActiveAdmins(excludeTarget)===0` ‚Üí reuse auth's `LastAdminError 409`) applies to ban + demote. NOTE the last-admin branch is effectively unreachable cross-actor (the acting admin is itself a counted active admin ‚âÝ target), but it's the spec's source of truth + the real guard for the self-delete path ‚Äî keep it. Promote has NO last-admin concern; a banned user CAN be promoted (`is_admin`/`banned` are independent).

- **`admin_actions` is append-only with RESTRICT FKs (Layer 9a).** `AdminActionRepository.record` is INSERT-only (no reads/updates/deletes from the app) ‚Äî the trail must survive for appeals (spec "Audit log", read directly from the DB; a `/admin/audit` UI is v1.1). Both FKs are RESTRICT (no `onDelete` rule) ‚Äî targets are soft-deleted/banned, never hard-deleted, so rows never orphan. `reason` is non-null for promote/demote/ban, `null` for unban (no reason modal). `action` is an app-validated TEXT enum (`promote|demote|ban|unban`), same convention as `Notification.type`.

- **Auth `UserRepository.setBanned` / `setAdmin` / `listForAdmin` are no-lock (Layer 9a).** Same Layer-6 convention as `updateProfile`/`markDeleted` ‚Äî the User aggregate has no concurrent mutators (admin actions are single-tab from the admin's side). `listForAdmin` EXCLUDES soft-deleted rows (a deleted account is gone; the spec status column is exactly active/banned) but INCLUDES banned (the table shows status + offers `[Unban]`); search matches name OR email case-insensitive; capped at 200 rows (no pagination island in 9a ‚Äî `// TODO(scale)`). The guards (last-admin, self-mod) are NEVER in the repository ‚Äî they live in the services.

- **Venue CRUD (Layer 9b) lives in `match_lifecycle`, NOT `moderation` ‚Äî and writes NO audit row.** Contrast with 9a: `BanUserService` is in `moderation` because it's cross-context (coordinates auth + match_lifecycle) AND audited. Venue create/update has neither property ‚Äî Venue is `match_lifecycle`'s own aggregate, the only other table it touches is its own `Match` (the guard count), and spec `/admin/venues` does not mandate `admin_actions` (only `/admin/users` + the optional later hide-log do). So `CreateVenueService` / `UpdateVenueService` sit in `match_lifecycle/application`, gated by `requireAdmin()` at the route (the service is auth-agnostic, exactly like `CreateMatchService`). The moderation `about.md` mention of "venue/match takedowns" refers to 9c hide/delete, not 9b CRUD. Don't add an `admin_actions` row for venue edits.
- **`CreateVenueService` generates the venue id app-side (`crypto.randomUUID`), overriding the DB `gen_random_uuid()` default.** This is the ONE place we pass an explicit `Venue.id` to Prisma `create` ‚Äî needed so `defaultCoverIdForVenue(id)` (the spec's deterministic-by-id cover formula, global.md ¬ß556) is reproducible BEFORE the INSERT. `UpdateVenueService` does NOT regenerate the id. The admin Add form leaves the cover picker on "Auto" ‚Üí omits `cover_id` ‚Üí the service applies the default; if the admin picks a swatch, that wins. Edit always sends `cover_id` (form prefilled).
- **The cover palette has two homes: domain owns the slugs, UI owns the pixels.** `src/match_lifecycle/domain/covers.ts` = `COVER_IDS` (the ~12 slugs, single source of truth) + `defaultCoverIdForVenue` + `isValidCoverId` (pure, reproducible on frontend/backend/SQL). `src/ui/lib/cover-style.ts` = `COVER_STYLES: Record<CoverId, {from,to,icon}>` (gradient + emoji) ‚Äî exhaustive over `COVER_IDS` so adding a slug without a style is a compile error. Cover hex values live ONLY in `cover-style.ts` (its own decorative palette, like `tokens.ts` for the brand) ‚Äî don't inline them elsewhere. Cover_id is stored on `Venue` + snapshotted onto `Match`, but the match hero / MatchCard still render a fixed placeholder gradient ‚Äî wiring `cover_id` into those renders is a deferred cosmetic pass, NOT done in 9b.
- **The venue deactivation guard lives in `UpdateVenueService`, fires ONLY on the active‚Üíinactive flip.** `existing.active && !patch.active` ‚Üí `countUpcomingNonCancelledAtVenue(id, now) > 0` (predicate: `start_time > now AND cancelled_at IS NULL`) ‚Üí `VenueHasUpcomingMatchesError 409` with `meta.upcomingMatchCount`. Reactivation (inactive‚Üíactive) and staying-active never run the count. The form mirrors it (toggle disabled + the exact spec hint when `upcomingMatchCount > 0`); `listAllForAdmin` decorates each row with that count via a filtered Prisma relation count (`_count.select.matches` with the where-clause) so the RSC has it without an N+1. No advisory lock on venue writes ‚Äî the Venue aggregate has no concurrent mutators (admin single-tab), same rationale as `UserRepository.updateProfile`.
- **`/admin` has its own four-tab `AdminBottomNav`; the public five-tab `BottomNav` is hidden there.** `src/ui/components/bottom-nav.tsx` HIDDEN_PATHS now includes `/admin` (alongside `/login`, `/welcome`). `SignedInChrome` already self-gates to `/my-matches` `/me` `/games`, so it never shows on `/admin` either. The admin nav (`app/admin/admin-bottom-nav.tsx`) links Users/Matches/Venues/Reports. Don't mount the public BottomNav inside the admin layout.

- **Admin hide-text is RSC-layer filtering + a no-lock `updateFlags` call (Layer 9c).** `PATCH /api/admin/matches/:id/hide-text` toggles `description_hidden` / `cancel_reason_hidden` on the Match row directly via `MatchRepository.updateFlags(id, flags)` ‚Äî no advisory lock (admin single-tab, these flags have no slot/status invariants). Available for ALL match statuses (content moderation ‚âÝ content editing ‚Äî spec personal.md "Hide text"). The original text is NEVER changed ‚Äî only the flag gate changes what the RSC renders. RSC moderation pattern: the page passes `match.cancelReason` straight through; the RSC flips it to `null` when `cancelReasonHidden === true` (same as `descriptionHidden`, same pattern as Layer 5/6.5). No audit row in v1 (spec: "add later"). **`updateFlags` returns `null` on P2025 (match not found) ‚Üí service throws `MatchNotFoundError`.**

- **Admin cancel bypasses `NotCaptainError` by passing the match's own `captainId` to `CancelMatchService` (Layer 9c).** `AdminCancelMatchService` does an unlocked pre-read of the match (`MatchRepository.findById(id)` ‚Äî no `tx`) to get `captainId` (immutable post-INSERT), then calls `CancelMatchService.execute({ matchId, captainId: match.captainId, cancelReason })`. The service re-reads the match under the advisory lock; the captain check passes because we supply the actual captain's id. All business rules (start-time guard, idempotency, mass-reject, watch wipe, notification fan-out) are UNCHANGED ‚Äî the admin just "impersonates" the captain for the auth check. Same bypass pattern used by `BanUserService` (also passes the target's own captainId). **Admin's identity lives only in `requireAdmin()` and the HTTP log ‚Äî never passed to the service.**

- **Admin edit uses a SEPARATE endpoint `PATCH /api/admin/matches/:id` and a SEPARATE RSC `/admin/matches/:id/edit` (Layer 9c).** The captain's `PATCH /api/matches/:id` route is UNCHANGED. The admin edit route reads `match.captainId` (unlocked pre-read, immutable field) and calls `EditMatchService.execute({ captainId: match.captainId, ... })` ‚Äî same bypass as admin cancel. `EditMatchForm` gained an optional `submitUrl?: string` prop (default `/api/matches/:id`); the admin edit RSC passes `/api/admin/matches/:id`. Both share the same client form, same OCC semantics (`updated_at` probe), same Zod whitelist schema. The captain edit page already had a comment: "Admin edit (Layer 9) will live at `/admin/matches/:id/edit`" ‚Äî honoured.

- **`admin_deleted` poll signal uses an ephemeral tombstone table, NOT `deriveMatchChange` (Layer 9c).** `admin_match_deletions(id, match_id TEXT no-FK, deleted_at, affected_user_ids UUID[])` is written BEFORE the physical DELETE (while JR/Watch rows still exist). `UpdatesStateService` queries it in parallel with the normal relationship scan (`findForUserSince`) and prepends `{ match_id, my_status: 'none', action: 'admin_deleted' }` entries to `matches_changed`. The 24h TTL is cleaned by `InboxTtlService` via `AdminMatchDeletionRepository.deleteOlderThan`. `MatchChangeAction` union includes `'admin_deleted'` but `deriveMatchChange()` NEVER emits it (a comment in the file says so). **`my_status: 'none'` is correct** ‚Äî JR rows are physically gone, no state to derive.

- **`WatchRepository.listForMatch` is now `tx?` (optional) as of Layer 9c.** Previously mandatory-`tx` (only called inside `withMatchLock`). Made optional to support the admin delete pre-scan (which reads watcher ids WITHOUT a lock, before the cascade delete). The adapter uses `tx ?? this.prisma` ‚Äî omit `tx` for unlocked reads, pass `tx` from `withMatchLock` for locked reads (notifyWatching callers unchanged). **Four no-lock write endpoints total now** (was three after Layer 6): same list + `DELETE /api/admin/matches/:id` (AdminDeleteMatchService ‚Äî no lock, terminal hard delete).

- **Admin hard delete has no notifications or emails (spec personal.md "Delete") (Layer 9c).** `AdminDeleteMatchService` does NOT call `CancelMatchService` (which would send notifications). It reads affected users ‚Üí writes tombstone ‚Üí calls `prisma.match.delete` directly (Prisma cascade handles children). The tombstone's only purpose is the global poll `admin_deleted` signal ‚Äî NOT inbox/email. No audit row in `admin_actions` (match delete is not a user-level action). The per-match poll `GET /api/matches/:id/state` returns 404 after hard delete ‚Üí the client redirects to `/games`.

- **`Report.target_match_id` is `onDelete: SetNull`, NOT Cascade (Layer 9d).** This is the deliberate resolution of a spec tension: reports are "retained indefinitely" (appeals/history) but matches CAN be admin hard-deleted (9c). A Cascade would erase the report trail; a RESTRICT would block the 9c delete with an FK violation. SetNull lets the delete proceed and leaves an ORPHANED report (`type='match'`, `target_match_id=NULL`) that survives. `ListAdminReportsService` groups orphans INDIVIDUALLY (key `${type}:orphan:${reportId}`, never collapsing them by a shared NULL) and renders "[Deleted match]" with only `[Dismiss]` available. `reporter_id` / `target_user_id` / `reviewed_by` stay RESTRICT (users are only soft-deleted). The Postgres UNIQUE treats NULLs as distinct, so orphans never collide on the dedup constraint.

- **Report status transitions: the per-modal rules (¬ß342/¬ß346) WIN over the summary (¬ß322) (Layer 9d).** The spec summary lists "Hide text" as a `[Review]` action that flips reports to Reviewed, but the detailed modal rules say hide-text toggles do NOT change report status. Implemented per the detailed rules: only **Ban / Cancel / Delete** flip ALL `new` reports on the target ‚Üí `reviewed` (`POST /api/admin/reports/mark-reviewed`); **Dismiss** flips ONLY the open report ‚Üí `dismissed` (`POST /api/admin/reports/:id/dismiss`); **hide-text changes nothing**. The "open report" the modal acts on = most-recent `new`, else most-recent overall (spec ¬ß321).

- **Review actions REUSE the existing admin endpoints; 9d adds only the status side-effect (Layer 9d, spec ¬ß348).** The `/admin/reports` Review sheet calls the SAME HTTP routes as `/admin/users` + `/admin/matches` (Ban ‚Üí 9a `/api/admin/users/:id/ban`, Cancel/Delete/Hide ‚Üí 9c routes). After a successful destructive action the CLIENT calls `mark-reviewed` (two-step, best-effort-but-awaited ‚Äî the action already committed if mark-reviewed fails). Do NOT build a `ResolveReportService` that re-implements ban/cancel/delete ‚Äî that would duplicate logic the spec explicitly says to reuse.

- **`SubmitReportService` 404s a banned/deleted player target for privacy parity (Layer 9d).** A new player report on a missing OR banned OR soft-deleted user ‚Üí `ReportTargetNotFoundError` (404) ‚Äî outside observers can't distinguish, mirroring the `/users/:id` sentinel. Match targets only 404 when truly absent (cancelled matches ARE reportable ‚Äî the page stays reachable). Self-report ‚Üí `CannotReportSelfError` (400). Dedup is silent: a repeat report returns 200 with no new row (`insertIfAbsent` ‚Üí `'duplicate'`), the route + modal show the same success toast either way (spec "no toast spam"). **No rate-limit implemented** ‚Äî like every other endpoint in this repo, the spec's per-user limit is deferred; the UNIQUE constraint is the only spam guard in v1.

- **`MatchRepository.findForAdminByIds` reuses the `AdminMatchRow` shape (Layer 9d).** Added so `ListAdminReportsService` resolves every match-report target's venue/status/hide-flags in ONE query (no N+1). The adapter's row‚Üí`AdminMatchRow` projection was extracted into a shared `adminRowToDomain` helper used by BOTH `findForAdmin` and `findForAdminByIds`. Status is derived on-read (`deriveMatchStatus` + `computeSlots`), same as `ListAdminMatchesService` ‚Äî never a DB column. When you add a port method, remember the FOUR test fakes (`tests/match_lifecycle/_helpers/fakes.ts` + three inline ones in create-match / list-discover / list-map tests) all `implements MatchRepository` and must gain the stub or tsc breaks every test that constructs one.

### Deployment (Layer 10 ‚Äî ADR-0006)

- **Production = Docker Compose on ONE VPS; images built in CI, pulled from GHCR.** `Dockerfile` (stages base/deps/build/runner/cron), `docker-compose.yml` (db + app + caddy + cron), `.github/workflows/deploy.yml`. Full topology + every rejected option in [ADR-0006](./docs/adr/0006-deploy-topology.md); the live bring-up steps are in [docs/deploy-runbook.md](./docs/deploy-runbook.md). Don't reach for a PaaS/Vercel pattern ‚Äî the target is the self-hosted box.
- **`next.config` `output:'standalone'` is load-bearing.** The runner copies `.next/standalone` + `.next/static` + `public` + an **explicit `node_modules/.prisma`** (the Prisma query engine is NOT auto-traced by Next ‚Äî verified during 10a). Don't delete that copy or switch off standalone.
- **Docker installs pnpm with `node-linker=hoisted`** (flat node_modules) so standalone tracing + the Prisma engine behave on the npm-style path; pnpm's default symlinked store isn't reliably traceable or copyable. The lockfile is linker-agnostic, so `--frozen-lockfile` still holds.
- **`NEXT_PUBLIC_*` are BUILD-time, not runtime.** `NEXT_PUBLIC_APP_URL` + `NEXT_PUBLIC_ABLY_SUBSCRIBE_KEY` are inlined by `next build` ‚Üí passed as `--build-arg` (CI + compose `build.args`). Setting them only in the runtime `environment` does nothing for the client bundle. Changing either needs a REBUILD. (The build stage also sets throwaway `DATABASE_URL`/`AUTH_*` so `env.ts` validation doesn't fail during route collection ‚Äî real values come at runtime.)
- **The Dockerfile's LAST stage is `cron`, so the `app` build pins `target: runner`.** Two images ship: slim `runner` (the app) and `cron` (source + deps + `tsx`). The slim runner can't run TypeScript, so the four Layer-7b crons live in the cron image. If you add a stage after `cron`, re-check every `target:` (compose app build + both CI build steps).
- **Migrations run in the container entrypoint.** `docker-entrypoint.sh` does `prisma migrate deploy` (idempotent, via a pinned global `prisma@6.19.3` CLI ‚Äî the slim bundle has no CLI) then `exec node server.js`, fail-fast. Compose orders app after a healthy `db`.
- **Compose profiles split the stack.** `caddy` is under `profiles:["prod"]` (gated by `COMPOSE_PROFILES=prod` in the VPS `.env`), `cron` under `profiles:["cron"]` (run-only ‚Äî `docker compose run --rm cron <cmd>`, never started by `up`). A local `docker compose up` (no profile) runs db+app only, app on localhost:3000 ‚Äî same file for dev smoke and prod.
- **Prod cron reuses `scripts/run-cron.ts` verbatim** (host crontab, `CRON_TZ=Europe/Prague`, DST-correct ‚Äî `deploy/crontab.example`). Its `process.loadEnvFile('.env.local')` is wrapped in try/catch so the same script works in the container (env injected by compose). The `cron` compose service carries `AUTH_*` ONLY because `env.ts` validates them at import, even though the crons never use them. Nightly `pg_dump` via `deploy/backup.sh`. **Still no cron HTTP route** (spec: cron is server-side) ‚Äî don't add one.
- **TLS = Cloudflare Origin cert + stock `caddy:2-alpine`, no ACME** (the orange cloud intercepts HTTP-01). Cert/key are mounted from `caddy/certs/` (git-ignored, `*.pem`/`*.key`). `AUTH_URL=https://pitchup.online` + the existing `trustHost:true` make Auth.js callbacks correct behind the proxy. DNS-01 auto-cert (custom Caddy build) is the documented alternative.
- **CI gate needs NO env** (unit tests don't import `env.ts` ‚Äî vitest doesn't load `.env.local`) but runs `pnpm exec prisma generate` before `tsc`. The deploy job is gated on `vars.DEPLOY_ENABLED=='true'` so CI stays green before the VPS exists. GHCR image refs must be **lowercase** (the workflow lowercases the owner).

## Before you change anything significant

1. **Identify the source of truth.** Spec for behavior, ARCHITECTURE for code patterns, CODING_STANDARDS for universal principles, mockups for visuals.
2. **Search for prior art.** Grep for related anchors (`ANCHOR:`), check `composition.ts` files, look at `about.md` of nearby bounded contexts.
3. **Update the spec first if behavior changes.** Then `app-map.md` if ERD/status table affected (the spec-file edit hook reminds you).
4. **Write an ADR for non-trivial decisions** (`docs/adr/NNNN-short-slug.md`). Context / Decision / Consequences.
5. **One bounded context per PR/commit when possible.** If the change spans contexts, the commit message names all of them: `refactor(match_lifecycle,notifications): ...`.

## Adding a new feature (workflow)

1. **Confirm the spec covers it.** If not ‚Äî stop, escalate in Russian to the human.
2. **Identify or create the bounded context.** New context = new `src/<name>/` folder + `about.md` + composition root.
3. **Domain first.** Entities, value objects, ports, domain errors.
4. **Use case next.** Service in `application/`, accepting ports via constructor. Tests with fakes.
5. **Adapter.** Prisma repository in `infrastructure/`. Integration test against real Postgres.
6. **Wire it.** Update `composition.ts`.
7. **Route handler.** Thin parse-validate-call-map in `app/api/.../route.ts`.
8. **UI.** Server Component page + client island where needed. Use existing UI kit components from `src/ui/`; new visuals ‚Üí add to `src/ui/` first, then use in the screen.
9. **Update AGENTS.md** if the workflow taught you a new gotcha.
